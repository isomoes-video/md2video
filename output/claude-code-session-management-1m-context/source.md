# 使用 Claude Code：会话管理与 1M 上下文

Claude 的这篇博客文章解释了：在 Claude Code 中，你如何管理会话、上下文和压缩，往往比很多人预期的更能决定最终结果。随着新 `/usage` 命令发布，用户反复提到一个很实际的问题：现在 Claude Code 已经拥有 100 万 token 的上下文窗口，人们到底该如何判断什么时候继续使用一个长会话，什么时候重新开始，以及什么时候该使用 compact、rewind 或 subagent 这类工具？

## 上下文、压缩与上下文腐化速览

上下文窗口就是模型在生成下一次回复时能够看到的全部内容。其中包括系统提示词、到目前为止的对话、每一次工具调用与工具输出，以及所有已经读取过的文件。Claude Code 现在提供 100 万 token 的上下文窗口。

更大的窗口确实很强大，但并没有消除权衡。随着窗口变大，性能可能下降，因为注意力会分散到更多 token 上，而过时的信息也会开始与当前任务争夺注意力。文章将这种现象称为上下文腐化。

当会话接近上下文硬上限时，Claude Code 会自动把当前会话总结成更短的描述，并基于这个摘要继续。这就是压缩。用户也可以手动执行压缩。

## 每一轮回复都是一个分叉点

当 Claude 完成一个步骤后，用户有多种继续方式：

- 在同一个会话里继续
- 使用 `/rewind` 或双击 Esc 回到更早的消息并重新尝试
- 使用 `/clear` 基于提炼后的任务说明开启新会话
- 使用 `/compact` 总结当前会话，再从摘要继续
- 使用 subagent，让下一段工作在它自己的干净上下文窗口里运行

文章强调，这些不只是便捷功能，更是上下文管理的控制手段。

## 什么时候应该开启新会话

Anthropic 的经验法则是：当你开始一个新任务时，也应该开启一个新会话。100 万 token 的上下文窗口确实支持更长的任务，但并不能消除上下文腐化。如果下一步仍然属于同一个任务，而且之前读取的文件或已经做出的决策依然相关，那么留在当前会话里通常更高效。如果任务已经真正发生变化，那么一个全新的会话往往更好。

## 与其纠错，不如回退

文章认为，当 Claude 已经沿着错误方向探索过之后，`rewind` 往往比直接补一句纠正更有效。如果 Claude 读对了文件，但走错了实现路线，那么回退可以保留有价值的准备工作，同时丢弃失败尝试。然后你可以基于刚刚得到的认识，从那个更早的节点重新提示它。

## 压缩，还是直接开启新会话？

`/compact` 和 `/clear` 都能减少上下文，但它们解决的问题并不相同。

- 使用 `/compact` 时，Claude 会自己总结当前会话，并从自己的摘要继续。这种方式更省力，也能自动保留不少有用细节，尤其是在你给出提示方向时。
- 使用 `/clear` 时，用户需要手动写新的任务说明，并从一个干净会话开始。这更费力，但最终保留下来的上下文完全由用户自己决定。

## 什么会导致糟糕的自动压缩？

文章指出，糟糕的自动压缩通常发生在这种情况下：会话先在某个方向上进行了很长时间的探索，而下一条提示却突然转向另一个边缘问题。由于模型是在上下文已经很拥挤的情况下执行压缩，那些对前一条主线来说不够核心的细节，就很可能在摘要中丢失。

## Subagent 与全新的上下文窗口

当某一段工作会产生大量中间输出，而主会话又没有必要保留这些内容时，subagent 就特别适合。一个 subagent 会获得自己的干净上下文窗口，独立完成工作，然后只把综合后的结果返回给主会话。Anthropic 提出的判断标准是：我之后还需要这些工具输出吗，还是只需要结论？

## 文章给出的实用结论

推荐的对应关系很直接：

- 还是同一个任务，而且上下文仍然相关：continue
- 走错分支了：rewind
- 会话变臃肿了，但任务没变：compact
- 真正开始了一个新任务：clear
- 工作过程很嘈杂，但你只关心结果：subagent

核心教训是：1M 上下文只有在被主动管理时，价值才最大。更大的上下文确实提供了更大的操作空间，但只有当用户有意识地塑造窗口中保留什么内容时，结果才会继续变好。
