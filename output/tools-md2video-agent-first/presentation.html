为什么 md2video 值得单独写一篇文章

不是传统应用的默认设定

  • 大多数软件仍然围绕人类操作者设计：GUI、CLI、SDK 各自服务不同入口
  • md2video 的第一输入却更像 prompt 文件，直接面向 agent 执行
  • 仓库本身预先声明了阶段、边界、产物目录与质量要求

HUMAN ROLE

给方向、审查中间产物、决定是否进入下一阶段

AGENT ROLE

读 prompt、调工具、写 artifact，并维护整个 workspace 一致性

所以它更像一个 agent workspace 样本，而不只是“把 Markdown 变视频”的脚本集合。

第一输入不是 GUI，而是一个可执行的 workspace 契约

Stage Prompts

每个 prompt 明确告诉 agent：当前阶段该产出什么、输出到哪里、遵守什么规则。

Output Workspace

所有成果都收敛到同一个 `output/<slug>/` 目录，天然可检查、可续跑、可组合。

Artifact Contract

`presentation.html`、`styles.css`、`script.json`、`audio/`、`video.mp4` 都是显式产物。

真正关键的不是“让 AI 帮我点按钮”，而是提前把任务表面、工具边界和输出契约都写进仓库。

Skills + MCP + Workspace，在同一个目录里协同

Prompt / Stage Files

计划、TTS、合成等 prompt 把行为拆成清晰阶段。

Skills

Reveal.js skill 提供结构、样式、审查和导出规则。

MCP / Scripts

TTS 与 FFmpeg 组合脚本负责真正执行。

Agent

不是从零猜流程，而是在完整工作表面上规划、调用、写入并验证。

Artifacts

`presentation.html` / `script.json` / `audio/*.mp3` / `video.mp4`

知识被 skill 打包

执行能力被脚本打包

workspace 把二者绑到同一上下文

从 GUI、CLI 到 CUI：聊天回路变成新的编排层

GUI

点选界面

适合直接操作具体控件，但不擅长承接长链路任务。

CLI

命令入口

适合精确调用单个动作，但意图编排仍然偏手工。

CUI

聊天编排层

用户表达目标，agent 决定该读哪些 prompt、调用哪些工具、产出哪些 artifact。

这并不是说 GUI 和 CLI 消失了，而是它们开始退到执行层；CUI 负责承接意图，repo 负责承接契约。

md2video 把 Markdown 变成视频，走的是可检查的文件系统流程

1

source.md 或直接文本输入，作为 planning 阶段上下文

2

plan prompt 生成 slides、theme、`script.json` 与 `output.pdf`

3

review 审查 workspace，确认是否进入语音阶段

4

tts prompt 为每页旁白写出 `audio/slide-XX.mp3`

5

combine prompt 按 PDF 页面顺序与音频时长合成 `video.mp4`

Markdown → slides → narration units → MP3 → MP4。每一跳都是清晰 artifact，不是隐藏在黑箱里的中间状态。

这个项目很小，但它在提示一种新的默认软件形态

Agent-native software sample

  • 有 prompt 入口

  • 有工具访问层
  • 有严格输出契约

背后的趋势

软件交付不再只是一款 app 或一个 library，也越来越像一个供 agent 执行、供人类审核的 workspace。

人表达意图，agent 围绕清晰定义的 artifact 去工作，这可能会成为很多 AI 工作流里的新默认值。
